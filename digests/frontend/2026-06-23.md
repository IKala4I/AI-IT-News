## 📱 React Native & Expo

### Reanimated 4.5.0 — CSS pseudo-selectors + SharedTransitionBoundary ⭐ Item of the week
Released June 23, this is the largest Reanimated drop since 4.0 stable. Key new capabilities:

**CSS pseudo-selector support**
`:hover`, `:active`, and `:focus` are now available as native selectors on iOS, Android, *and* web. The native implementations mean these respond to actual pointer/touch events rather than JS-simulated state — no extra `useState` or `Pressable` state tracking needed for simple hover/focus styles.

```js
// before: useState + Pressable
// after:
const style = useAnimatedStyle({ ':hover': { opacity: 0.8 } });
```

**SharedTransitionBoundary component**
Scopes shared element transitions to a subtree so that unrelated screens don't bleed animation state into each other during navigation. Previously, shared transitions applied globally and caused visual glitches when multiple navigators were mounted simultaneously.

**SVG animation on web**
`react-native-svg` components can now be animated on web via Reanimated's CSS animation engine, closing a long-standing gap between mobile and web parity.

**iOS Core Animation for shadow/border/background**
When the `IOS_CSS_CORE_ANIMATION` feature flag is enabled, shadow, border, and background CSS transitions now run on Core Animation layers instead of the JS loop — native-speed for the properties that were previously the slowest to animate on iOS.

**30+ bug fixes** covering iOS release-build layout animations, Android ANR deadlocks, memory leaks on unmounting, and CSS interpolation.

**Breaking changes:** None reported; this is a minor version bump in the 4.x series.

- [GitHub release — v4.5.0](https://github.com/software-mansion/react-native-reanimated/releases/tag/4.5.0)

## ▲ Next.js / Vercel

### Next.js canary.61 + .62 — Cached Navigations goes global (June 23)

**canary.61** (June 23, 00:10 UTC) is the more significant of the two:
- **Global opt-in for Cached Navigations runtime stage.** The experimental `cachedNavigations` flag can now be set globally in `next.config.js` rather than per-route. This means client-side navigations reuse the already-rendered Server Component payload rather than re-fetching, eliminating round-trips for navigations to previously visited routes.
- **Removed legacy Partial Pre-Rendering codepaths.** The old PPR implementation that predated Cache Components has been deleted from the codebase. If you were relying on undocumented internal PPR behavior, canary.61 is a natural checkpoint to validate.
- React experimental channel upgrade included.

**canary.62** (June 23, 15:22 UTC):
- Fixed a false-positive in the `export const dynamic` detection inside Cache Components — Next.js was incorrectly flagging dynamic exports inside cached subtrees as an error.
- Turbopack improvements.

The stable track is **16.2.x**; these are pre-release only.

**Why it matters:** Cached Navigations + Cache Components together form the new Next.js caching model that replaces the old `fetch`-based data cache and `getStaticProps`/`getServerSideProps` mental model. These canaries are how to track its shape before it stabilises in 16.3.

- [Next.js 16.3 canary overview — buildwithmatija.com](https://www.buildwithmatija.com/blog/nextjs-16-3-canary-cache-components-turbopack)

### Vercel Flags — platform-native feature flags, blog post (June 22)

Vercel published a dedicated blog post for **Vercel Flags**, the feature flag service built directly into the Vercel platform (announced at Vercel Ship June 17 but detailed further here). Key details:

- **Server-side evaluation only** — flags are evaluated at the Edge/server before the response is sent. No client-side JS execution means zero CLS or FID impact from flag evaluation.
- **Flags SDK updates** this month: bulk flag evaluation, jittered ingest retries, OIDC authentication support for flag clients, hot-path evaluation caching.
- Works alongside third-party providers (LaunchDarkly, Statsig, etc.) via the Flags SDK adapter pattern, so you don't have to migrate your existing flag system — you just gain the Vercel Toolbar integration and Analytics observability on top.

**Why it matters for Next.js teams:** The Flags Toolbar lets you override flag state locally without touching code or affecting other users — a developer experience win for feature development and QA.

- [Vercel Flags docs](https://vercel.com/docs/flags)
- [Releasebot — Vercel Flags SDK June 2026](https://releasebot.io/updates/vercel/flags)
