### React Compiler tradeoffs become a live debate

Two independent pieces this week push back on the "just turn on the Compiler" narrative:
- A widely-shared critique (via TWIR #288) argues React Compiler is "a retrofit" bolted onto a runtime that wasn't designed for ahead-of-time compilation, and can *increase* bundle size in some cases.
- Isaac Okoro (LogRocket), via React Status #481, documents concrete cases where relying on the Compiler's auto-memoization broke existing behavior ŌĆö a useful companion piece to the critique above.

Counterpoint: **Rsbuild 2.1**'s Rust port of the Compiler (already covered June 26) is being cited as claiming **7ŌĆō13x speedups** over the Babel-based version, which softens the performance side of the argument even as the correctness/bundle-size side gets more scrutiny.

**Why it matters:** If you've been compiling-by-default on recent React versions, both pieces are worth a read before your next audit ŌĆö the Compiler is not yet a fully "invisible" optimization.

- [This Week In React #288](https://thisweekinreact.com/newsletter/288)
- [React Status #481](https://react.statuscode.com/issues/481)

### Expo SDK 57 (June 30) ŌŁÉŌŁÉ

SDK 57 bumps React Native **0.85 ŌåÆ 0.86** while holding React at 19.2 (unchanged from SDK 56). Per the React Native team, 0.86 has **no breaking changes from 0.85** ŌĆö Expo skipped its usual beta phase as a result and shipped straight to stable, a sign the SDK release cadence is loosening from three-a-year toward smaller, more frequent, non-breaking drops (mirroring RN's own shift to bi-annual non-breaking releases).

**Highlights:**
- Improved edge-to-edge Android support; light/dark mode emulation in DevTools
- `expo-dev-client`: choose auto-launch of recent projects vs. showing the launcher UI (iOS)
- `expo prebuild` now clears/regenerates native directories **by default** (previously opt-in)
- `expo-image` adds cache seeding/reading via `writeToCacheAsync`/`readFromCacheAsync`
- `expo-router` expands badge support (header placements, Android toolbar menu icons)
- reanimated/worklets/gesture-handler bumped in lockstep

**Migration steps (explicit):**
```bash
npx expo install expo@^57.0.0 --fix
npx expo-doctor@latest
```
If you use Continuous Native Generation, delete native directories first (prebuild regenerates them by default now); otherwise run `npx pod-install`. Development-build users should rebuild; Expo Go users should consider migrating to dev builds.

**Known issue:** Android memory usage may rise **25ŌĆō30%** when using `react-native-reanimated` with Hermes V1, caused by compiler changes introduced back in RN 0.85 ŌĆö worth checking if your app leans heavily on Reanimated.

- [expo.dev/changelog/sdk-57](https://expo.dev/changelog/sdk-57)

### Next.js 16.3 canary progress ŌĆö Instant Navigation fixes (canary.72ŌĆō78, June 30 ŌĆō July 4)

Continued hardening of the Instant Navigations feature (announced June 25, covered last week) ahead of stable. Notable canary items:

### Vercel: Run any Dockerfile on Vercel (June 30)

Add a `Dockerfile.vercel` and deploy any containerized app ŌĆö Go, Rails, Spring Boot, Express, Laravel, whatever ŌĆö onto Vercel's Fluid compute, billed by actual CPU usage rather than reserved capacity, with no local daemon or registry management.

**Why it matters for React/Next.js teams:** non-Node backends and legacy services can now sit alongside a Next.js frontend on the same platform with unified previews, instead of requiring a separate host.

**Breaking changes:** None ŌĆö opt in by adding one file.

- [vercel.com/blog/dockerfile-on-vercel](https://vercel.com/blog/dockerfile-on-vercel)

### Vercel Services: run full-stack apps in one project (June 30)

New capability to run multiple frameworks/services (frontend + backend) inside a single Vercel Project with atomic deployments, shared preview environments, and private service-to-service bindings ŌĆö no public internet hop, no CORS/reverse-proxy plumbing. Zero-config detection for FastAPI, Flask, Express, Hono, Go, Rust; integrates with Queues, Workflows, Cron, and Sandbox. `vercel dev` now gives production-like local dev across services.

**Breaking changes:** None; opt in via a new `services` key in `vercel.json`.

- [vercel.com/blog/vercel-services-run-full-stack-on-vercel](https://vercel.com/blog/vercel-services-run-full-stack-on-vercel)

### Vercel and Shopify are rebuilding Hydrogen (June 30)

Shopify's headless commerce framework Hydrogen is being rebuilt as open-source and **runtime-agnostic** ŌĆö usable with Svelte, Nuxt, Next.js, or a custom framework, removing the prior React/Remix-only lock-in. New architecture: a shared-JS Core layer (Shopify API calls), a Client layer of pre-built React state components (cart handling via `createCartComponents()`), and a Server layer built on native Next.js caching/revalidation instead of proprietary abstractions.

**Why it matters:** Next.js-based commerce sites get first-class, framework-native Shopify cart/checkout integration instead of a separate Hydrogen-specific runtime.

**Migration:** Ships as a preview branch on GitHub, not yet stable ŌĆö no forced migration for existing Hydrogen users.

- [vercel.com/blog/vercel-and-shopify-are-rebuilding-hydrogen](https://vercel.com/blog/vercel-and-shopify-are-rebuilding-hydrogen)

### Realtime voice agents on AI Gateway (June 29)

Vercel's AI Gateway adds audio/voice support (two-way realtime voice, TTS, STT) via OpenAI and xAI models, available now in AI SDK 7 (beta) through a new `useRealtime` hook that manages WebSocket connections, mic access, and audio playback, with short-lived auth tokens to avoid exposing API keys client-side.

**Why it matters:** React apps already on AI SDK 7 can add conversational voice UI using the same call patterns as text/image/video generation ŌĆö no separate STT/LLM/TTS integration needed.

- [vercel.com/blog/realtime-voice-agents-on-ai-gateway](https://vercel.com/blog/realtime-voice-agents-on-ai-gateway)

### shadcn/ui: Base UI becomes the default component library ŌŁÉŌŁÉ

The biggest ecosystem-direction story of the week (shadcn@4.13.0, published July 3). New projects created via `npx shadcn init` now default to **Base UI** (stable, 6M+ weekly downloads) instead of Radix Primitives; the shadcn docs' component pages now open on the Base UI tab by default, and shadcn's own telemetry shows new-project creators picking Base UI over Radix roughly 2:1.

**Not a breaking change:** Radix is explicitly not deprecated ŌĆö existing Radix-based projects need no changes, and `npx shadcn init -b radix` still opts new projects into Radix. A new **migration skill** (not a codemod) does progressive, type-checked, component-by-component migration with per-component reports and clean git history for teams who do want to move.

Related: `@shadcn/react@0.2.0` (June 30) continues iterating on the new headless/unstyled React package first introduced as 0.1.0 the prior week.

**Why it matters:** if you scaffold new UI with shadcn, expect Base UI primitives under the hood going forward ŌĆö check any code that currently assumes Radix's API surface.

- [ui.shadcn.com/docs/changelog/2026-07-base-ui-default](https://ui.shadcn.com/docs/changelog/2026-07-base-ui-default)
