## 📱 React Native & Expo

### React Native 0.83.10 — iOS binary caching, layout & DevTools fixes (June 26)

Patch release; no new features, all backport fixes for the 0.83 stable line.

**What changed:**

| Area | Fix |
|---|---|
| **iOS build performance** | Prebuilt Hermes, ReactNativeDependencies, and ReactNativeCore tarballs are now cached in `~/Library/Caches/ReactNative` — clean installs across projects no longer re-download these large binaries |
| **Layout** | Fixed Yoga node ownership bug when `display: contents` is used inside absolutely positioned subtrees; corrected `hasNewLayout` flag propagation |
| **DevTools** | Fixed false "multi-host" detection for apps that use additional Hermes runtimes (e.g. Reanimated) |
| **Android** | Corrected `fetch()` response URL after redirects; capped WebSocket queue size to prevent inspector disconnections on large payloads |
| **iOS stability** | Fixed crash when toggling `removeClippedSubviews`; made Podfile.lock checksums deterministic across machines |

**Why it matters:** The iOS binary cache alone can shave minutes off CI cold-start times for teams that run `pod install --clean-all` on every build.

**Breaking changes:** None.

- [GitHub release — RN 0.83.10](https://github.com/facebook/react-native/releases/tag/v0.83.10)

## ▲ Next.js / Vercel

### Next.js 16.3: Instant Navigations — official blog post (June 25) ⭐⭐

This is the week's headline story for Next.js developers. The core team (Andrew Clark, Josh Story) published a detailed blog post on **Instant Navigations**, a suite of opt-in features that bring SPA-style link responsiveness to the server-driven Next.js model.

**The problem being solved:** In a server-components app, clicking a link today causes a blank pause until the server responds. SPAs feel instant because they ship a per-route JS bundle the client can render immediately. Next.js 16.3 borrows the SPA trick without giving up server rendering.

#### How it works

Enable the two config flags:

```ts
// next.config.ts
const nextConfig = {
  cacheComponents: true,     // enables Cache Components (replaces legacy PPR)
  partialPrefetching: true,  // enables per-route shell prefetching
};
```

For each route that awaits data, you now choose one of three modes:

- **Stream** (wrap slow parts in `<Suspense>`) — user sees an instant loading shell; data streams in.
- **Cache** (use `'use cache'`) — user sees previously-cached UI instantly; re-validated in background.
- **Block** (`export const instant = false`) — deliberately opts out; navigation waits for server (e.g. blog posts where a loading skeleton would be weird).

#### Prefetching redesign

Instead of one prefetch request per link in the viewport (the current behavior, which fires dozens of requests on scroll), 16.3 prefetches **one reusable shell per distinct route**. A chat sidebar with 30 links now fires one shell request for `/chat/[id]` rather than 30.

`<Link prefetch={true}>` opts into per-link prefetching on top of the shell baseline. `export const prefetch = 'allow-runtime'` extends it to request-time cached content.

#### Developer tooling

- **Instant Insights panel** in Next.js DevTools — surfaces blocking routes as errors in dev. (Safari has known issues; use Chrome/Firefox.)
- **Navigation Inspector** — lets you pause every navigation at its shell and visually inspect what's prefetched.
- **`instant()` Playwright helper** — assert what's visible before the network response arrives:

```ts
import { instant } from '@next/playwright';

test('shell is instant', async ({ page }) => {
  await page.goto('/products/shoes');
  await instant(page, async () => {
    await page.click('a[href="/products/hats"]');
    await expect(page.locator('h1')).toContainText('Baseball Cap');
  });
});
```

**Real-world numbers (v0.app):** Navigation times dropped from several hundred ms to near-zero for most routes once Cache Components + Partial Prefetching were enabled.

**Try it:**
```bash
npm install next@preview
```

**Known issues (pre-stable):** Some blocking routes aren't caught by Instant Insights when Partial Prefetching is on; Safari has tooling glitches.

## 🧰 Libraries & Tooling

### Vite 8.1 — Bundled Dev Mode (released June 23, newsletter coverage June 26) ⭐

Vite 8.1 shipped just before this window but got its widest coverage this week via React Status and the Linear blog.

**Experimental Bundled Dev Mode** (`bundledDevMode: true`) bundles files during development the same way production does. Results from testing:
- **10,000-component benchmark:** ~15x faster startup, ~10x faster full page reloads
- **Linear (real production app):** cold start rendering 3x faster, full reloads 40% faster, 10x fewer network requests

Note: third-party plugins may have compatibility issues with bundled dev mode.

**Other 8.1 highlights:**
- **Experimental Chunk Import Map** — prevents hash cascading across chunks, improving cache efficiency
- **WebAssembly ESM integration** — `import foo from './lib.wasm'` with exported function access
- **Lightning CSS** — two missing features added; Lightning CSS is on track to become the default CSS processor in a future major

**v7.3.6** (June 25) is a patch for the v7 branch.

**Rolldown note (via React Status):** The Rolldown/Vite team temporarily removed React Compiler integration due to a 17% binary size increase — being re-evaluated.

- [Vite 8.1 announcement](https://vite.dev/blog/announcing-vite8-1)

### Rsbuild 2.1 — Rust React Compiler (June 26)

Rsbuild 2.1.0 (with 2.1.1 patch on June 27) ships native React Compiler support via Rspack's `builtin:swc-loader`:

```ts
// rsbuild.config.ts
export default {
  plugins: [pluginReact({ reactCompiler: true })],
};
```

Running the compiler in Rust (rather than the Babel JS pipeline) eliminates the JS-to-Rust serialization overhead that made the Babel-based approach slow on large codebases.

**Other 2.1 highlights:**
- Source imports enabled by default
- Parallel processing for SVGR and Babel plugins
- Vue, Sass, Less plugins converted to pure ESM
- Rspack 2.1.0 integration
- TanStack Start support

Requires `@rsbuild/core >= 2.1.0` and `@rsbuild/plugin-react >= 2.1.0`.

- [Rsbuild releases](https://github.com/web-infra-dev/rsbuild/releases)

### Linear's styled-components → StyleX migration

A detailed post-mortem by Andreas Skovhus (Linear engineer) on migrating ~100k lines of Linear's production codebase from styled-components to StyleX using an AI-built codemod.

**Results:** ~30% faster renders when navigating between pages.

**Why it matters:** Linear is a benchmark-quality web app built on React. A verified 30% navigation-render speedup from a CSS-in-JS swap is a meaningful data point for teams still on styled-components or Emotion who are evaluating the move.

The team used an agent-generated codemod rather than a hand-written script — a pattern increasingly common for large-scale style migrations.

### ForesightJS 4 — `<Foresight>` React component

ForesightJS predicts user intent from mouse movement and keyboard cues (hover trajectory, tab focus) and fires prefetch callbacks *before* the user actually clicks. Version 4 adds a first-party `<Foresight>` React wrapper:

```tsx
<Foresight onForesight={() => prefetchRoute('/dashboard')}>
  <Link href="/dashboard">Dashboard</Link>
</Foresight>
```

Works with any router; pairs naturally with Next.js `<Link prefetch>` and React Router's `prefetch` intent API.

- [foresightjs.com](https://foresightjs.com/)
