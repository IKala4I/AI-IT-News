### Simon Willison: AI Agents Should Never Be the "Directly Responsible Individual"

Willison published a short essay examining the "Directly Responsible Individual" (DRI) concept — the person ultimately accountable for an initiative's success or failure, used at Apple, GitLab, and elsewhere — arguing AI agents should never be assigned as a DRI no matter how autonomous they become, because accountability requires a human who can actually own the consequences. He cites a 1979 IBM training slide: "A computer can never be held accountable, therefore a computer must never make a management decision."

**Why it matters:** A concise, widely-shared framing for a problem many teams are hitting right now as they hand more workflow ownership to coding and agentic AI tools — useful language for org design as agents get more autonomy over real decisions.

Source: [simonwillison.net](https://simonwillison.net/2026/Jul/12/directly-responsible-individuals/)

## 💻 IT / General Tech News

### xAI's Grok Build CLI Caught Silently Uploading Entire Codebases — Including Secrets — to xAI's Servers

Independent packet-capture analysis of xAI's Grok Build CLI (v0.2.93) found it runs a hidden second data channel that bundles a user's entire git repository — including untouched files and `.env` secrets — and uploads it to a Google Cloud Storage bucket on every session; one test moved 5.1GB in 73 chunks from a 12GB repo. The tool's "Improve the model" opt-out toggle only affects training use, not the upload itself.

**Why it matters:** A serious trust and security issue for anyone using AI coding CLIs — credentials and proprietary code can leave the machine even when the agent never touches those files. Expect enterprise security teams to reassess or block Grok Build until xAI responds.

Source: [Security researcher writeup](https://gist.github.com/cereblab/dc9a40bc26120f4540e4e09b75ffb547)

### Compromised jscrambler npm Package Used to Drop a Cross-Platform Infostealer

Attackers used a hijacked publishing credential to push five malicious versions of the popular `jscrambler` npm package (8.14.0–8.20.0) over about three hours. Early versions ran a Rust-based infostealer ("IronWorm," linked to the Shai-Hulud malware lineage) via a preinstall hook; later versions moved the dropper into the package's main code so even `npm install --ignore-scripts` didn't block it. The malware harvests AWS/Azure/GCP credentials, crypto wallets, and Bitwarden vaults. Socket flagged it within six minutes of publication.

**Why it matters:** A security vendor's own package became a supply-chain attack vector — a reminder that even trusted dependencies need runtime scrutiny. Teams using jscrambler should audit for the affected versions immediately.

Source: [The Hacker News](https://thehackernews.com/2026/07/compromised-jscrambler-8140-npm-release.html)

### "GhostLock": 15-Year-Old Linux Kernel Flaw Enables Near-Certain Root Escalation

CVE-2026-43499 ("GhostLock"), a stack use-after-free in the Linux kernel's real-time mutex/futex priority-inheritance code, has been present in every mainline kernel since 2.6.39 (2011) until this month's fix. An unprivileged local attacker using only ordinary threading syscalls can trigger it for a roughly 97%-reliable root privilege escalation or container escape. The researchers who found it (Nebula Security's VEGA team) won $92,337 via Google's kernelCTF; distro patch rollout (AlmaLinux, CloudLinux, and others) is still ongoing.

**Why it matters:** Affects effectively all Linux distributions, Android, and embedded/IoT devices running affected kernels — a live, high-severity patching priority for sysadmins everywhere.

Source: [Nebula Security research](https://nebusec.ai/research/ionstack-part-2/)
