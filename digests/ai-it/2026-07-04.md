## 🤖 AI Models & Research

### Current AI Publishes a Data-Backed "Gap Map" of the Entire Open-Source AI Ecosystem

Current AI — a nonprofit "public option for AI" initiative backed by $400M in committed funding — released an interactive Gap Map on July 3 that catalogs the open-source AI landscape in depth: 421 profiled products (266 software tools, 85 models, 50 datasets, 20 hardware projects) from 228 organizations, sorted into 14 categories, winnowed down from an initial pool of over 24,000 candidate projects. Unlike a typical research report, they published the entire underlying dataset — 1,184 files under an MIT license on GitHub — so anyone can explore or build on it.

**Why it matters:** It's the first systematic attempt to show exactly where open-source AI is strong (inference tooling) versus where real gaps remain (independent compute, high-quality open datasets) — a practical resource for policymakers and funders trying to avoid a handful of closed labs controlling the entire AI stack.

Sources: [Simon Willison](https://simonwillison.net/2026/Jul/3/open-source-ai-gap-map/) · [Current AI Gap Map](https://map.currentai.org/)

### OpenAI Floats Giving the US Government a 5% Equity Stake

### Nvidia Launches Revenue-Sharing GPU Financing for Capital-Constrained AI Startups

Nvidia unveiled a new financing structure on July 1 letting AI cloud startups access GPUs now and pay later through a cut of future usage revenue rather than upfront cash. First partners are Sharon AI (up to 40,000 GPUs) and Firmus Technologies (a 360MW Indonesia data center, up to 170,000 GPUs) — a combined ~210,000-GPU, 432MW commitment. Nvidia now earns both its normal hardware margin and a recurring revenue cut on the same deployed chips.

**Why it matters:** It's Nvidia shifting from pure equipment vendor to financier-with-upside in its own customers' success — effectively vendor-financing the global AI buildout, which some analysts flag as adding circularity risk to an already capital-intensive industry.

## 💻 IT / General Tech News

### Google and FBI Take Down NetNut, a 2-Million-Device Residential Proxy Botnet

Google's Threat Intelligence Group, working with the FBI, Lumen Technologies, and the Shadowserver Foundation, disrupted "NetNut" (aka "Popa"), a residential proxy network built from at least 2 million compromised home devices — Android phones, smart TVs, and streaming boxes infected via trojanized or pre-installed malicious apps. Google observed over 300 distinct threat clusters using the network to route attacks through legitimate residential IPs and evade detection. The FBI seized the netnut.com domain, and Google disabled the backend infrastructure and pushed Play Protect blocks for the malicious apps.

**Why it matters:** Residential proxy networks are core infrastructure for ransomware and password-spray attackers looking to blend in with legitimate traffic — this takedown removes a widely resold piece of criminal infrastructure at real scale.

Sources: [BleepingComputer](https://www.bleepingcomputer.com/news/security/netnut-proxy-network-disrupted-2-million-infected-devices-cut-off/) · [The Hacker News](https://thehackernews.com/2026/07/google-disrupts-netnut-residential.html)

### FortiBleed Credential-Theft Campaign Directly Linked to Two Active Ransomware Gangs

Researchers uncovered that "FortiBleed" — a mass credential-harvesting operation using a custom packet-sniffing tool on compromised Fortinet FortiGate firewalls to intercept VPN logins — is directly tied to the INC Ransom and Lynx ransomware operations. The campaign targeted over 430,000 FortiGate devices worldwide, with roughly 19,000 having sniffers actively deployed. Investigators traced at least 12 ransomware deployments back to credentials stolen this way.

**Why it matters:** It's the first documented direct link between a mass firewall credential-theft campaign and downstream ransomware deployment. Any organization running FortiGate appliances should treat VPN credentials as potentially compromised and rotate them now.

Sources: [The Hacker News](https://thehackernews.com/2026/07/fortibleed-credential-theft-linked-to.html) · [BleepingComputer](https://www.bleepingcomputer.com/news/security/fortibleed-credential-theft-campaign-linked-to-lynx-ransomware/)

### New "Avalon" Malware Framework Bundles Ransomware, Credential Theft, and AI-Assisted Evasion

Researchers documented a previously unknown malware framework called Avalon, delivered via spoofed legal-document phishing emails leading to password-protected archives. It combines credential harvesting (browsers, crypto wallets, VPN/SSH/RDP, Windows Credential Manager), lateral movement, backup destruction, and a ransomware component called CrownX, with specific evasion routines targeting Defender, CrowdStrike, SentinelOne, and Sophos. Researchers flagged signs the framework was AI-assisted in its development.

**Why it matters:** It illustrates how sophisticated, all-in-one attack tooling is becoming accessible to less-skilled threat actors — raising the baseline threat level defenders need to plan for.

Source: [The Hacker News](https://thehackernews.com/2026/07/new-avalon-malware-framework-packs.html)

### Virginia's Ban on Selling Precise Geolocation Data Takes Effect

An amendment to Virginia's Consumer Data Protection Act took effect July 1, banning outright the sale of consumers' precise geolocation data (location within roughly 1,750 feet), regardless of what notice or consent language a company provides. Virginia becomes the third US state, after Maryland and Oregon, with such a ban; California, Massachusetts, Vermont, and Washington are pursuing similar bills.

**Why it matters:** Precise location data sales feed a large, largely unregulated data-broker industry — state-level bans like this are becoming a real compliance burden for ad-tech companies operating nationally, and momentum is building for more states to follow.

Sources: [The Record](https://therecord.media/virginia-enacts-ban-on-precise-geolocation-data) · [EPIC](https://epic.org/virginia-governor-signs-bill-banning-sale-of-precise-location-data/)

## 🛠️ New Tools & Releases

### VS Code 1.127 Gives AI Coding Agents Built-In Browser Control, Adds Extension Update Delay

Microsoft's July 1 VS Code release moves "browser tools" out of preview to general availability — AI coding agents can now independently open web pages, take screenshots, and click through a running app to verify their own work, without an external MCP server. It also adds per-site browser permission prompts and, notably, a mandatory two-hour delay before extensions auto-update to a newly published version (except from trusted publishers), meant to blunt supply-chain attacks via compromised extension updates.

**Why it matters:** It closes a real gap in agentic coding workflows — agents can now self-verify UI changes visually instead of relying only on tests — and the extension-update delay is a direct, practical response to a wave of malicious VS Code extension incidents.

Source: [VS Code Updates](https://code.visualstudio.com/updates/v1_127)

### GitHub Issue Fields Reaches General Availability

GitHub made "Issue Fields" generally available on July 2 across Free, Team, Enterprise, and GitHub Enterprise Cloud plans. It adds typed, structured metadata to issues (Priority, Effort, Start date, Target date by default), with values visible directly in issue lists and integration with AI tools via GitHub's MCP server. Over 40,000 organizations adopted it during the May preview.

**Why it matters:** Structured issue metadata that AI agents can read and write via MCP is a building block for more autonomous project-management workflows — coding agents can now triage by priority and effort directly.

Source: [GitHub Changelog](https://github.blog/changelog/2026-07-02-issue-fields-are-now-generally-available/)

### GitHub Copilot CLI No Longer Needs a Personal Access Token in GitHub Actions

GitHub shipped a change on July 2 letting Copilot CLI authenticate inside GitHub Actions workflows using the built-in, short-lived `GITHUB_TOKEN` instead of a manually created personal access token. Workflows just need the `copilot-requests: write` permission and an org policy toggle enabled.

**Why it matters:** It removes a real security footgun — long-lived PATs stored as CI secrets — for teams that automate Copilot CLI tasks in their pipelines.

Source: [GitHub Changelog](https://github.blog/changelog/2026-07-02-copilot-cli-no-longer-needs-a-personal-access-token-in-github-actions/)
